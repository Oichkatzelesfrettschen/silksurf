# sccache -- Rust Build Caching

## What sccache does

sccache wraps the Rust compiler. Every `rustc` invocation is intercepted: its
inputs (source hash, flags, rustc version, environment) are hashed into a cache
key. On a cache hit the cached object is returned without running the compiler.
On a miss the compiler runs and the output is stored for next time.

Practical impact on this workspace:
- First build after a clean: identical to uncached (primes the cache, ~15-20 min).
- Subsequent builds touching one crate: cache hit rate ~85-95%; rebuild time
  drops to seconds rather than minutes.
- After `cargo clean`: full cache miss (target/ gone) but sccache still hits
  (object files come from sccache, not target/); effective rebuild is fast.

## Installation

```sh
# Arch Linux (preferred, system package, no cargo needed):
sudo pacman -S sccache

# Any platform via cargo:
cargo install sccache --locked
```

Verify: `sccache --version`

## Configuration

### Workspace integration (.cargo/config.toml)

The workspace `.cargo/config.toml` already contains:

```toml
[build]
rustc-wrapper = "sccache"

[env]
CARGO_INCREMENTAL = { value = "0", force = false }
```

This wires sccache automatically for every `cargo build`, `cargo test`,
`cargo clippy`, and `cargo bench` invocation in this workspace.

No manual `RUSTC_WRAPPER=sccache` export is needed.

### Cache storage (~/.config/sccache/config)

The user-level sccache config is at `~/.config/sccache/config`. The default
generated by the install step sets 15 GiB at `~/.cache/sccache`. See that file
for tuning knobs.

### Checking cache health

```sh
sccache --show-stats
```

Key metrics to watch:
- `Cache hits rate (Rust)`: should be >= 80% on incremental builds.
- `Cache write errors`: should be 0; non-zero means disk full or permission issue.
- `Non-cacheable compilations`: proc-macro crates always appear here; normal.

Zero statistics between runs: `sccache --zero-stats`

### Cache invalidation triggers

Any change to the following invalidates ALL cached units for affected crates:
- Rust source files (`.rs`)
- `RUSTFLAGS` / `CARGO_ENCODED_RUSTFLAGS`
- rustc version (`rustup update` bumps this)
- Cargo feature flags passed at build time

The `Makefile` sets `RUSTFLAGS='-D warnings'` consistently for all targets.
Do not add ad-hoc per-invocation flags; they create separate cache entries
and defeat the cache.

## Bypassing sccache

For a single build without the cache (e.g. to measure true compile time):

```sh
RUSTC_WRAPPER="" cargo build --workspace --release
```

This overrides the cargo config for that one invocation only.

## Known limitations

| Limitation | Reason | Workaround |
|---|---|---|
| proc-macro crates always miss | sccache excludes `crate-type = proc-macro` invocations | None; expected behavior |
| Incremental compilation disabled | Incremental state is per-machine session, not content-addressed | Use sccache instead (equivalent speed for partial rebuilds) |
| RUSTFLAGS variation | Different flags = different cache key | Keep flags consistent; Makefile does this |
| Distributed builds | sccache supports Redis/S3/GCS backends for shared team caches | See sccache docs for backend config; update `~/.config/sccache/config` |

## Rust caching best practices (research summary)

These practices are applied in this workspace:

1. **Use sccache as `rustc-wrapper`, not `RUSTC_WRAPPER` in shell rc.** The
   cargo config approach is per-workspace, version-controlled, and applies
   consistently to `cargo clippy`, `cargo test`, etc. Shell rc is global and
   easy to forget.

2. **Disable incremental compilation when using sccache.** Incremental state
   is stored in `target/` as `.d` files and session-specific incremental
   indices. sccache does not cache incremental artifacts. Running both
   simultaneously causes partial cache misses and may produce stale artifacts
   after a `sccache --stop-server`. `CARGO_INCREMENTAL=0` is set in
   `.cargo/config.toml`.

3. **Keep RUSTFLAGS consistent across build invocations.** The hash key
   includes all compiler flags. Mixing `-D warnings` builds (Makefile) with
   no-flags builds (bare `cargo build`) creates two cache families, halving
   effective hit rate. All Makefile targets use `RUSTFLAGS='-D warnings'`.

4. **Do not set `codegen-units = 1` in `[profile.dev]`.** A single codegen
   unit forces rustc to emit one object file per crate; sccache caches at
   the per-crate level, so single-unit builds are cacheable. However,
   `codegen-units = 1` prevents parallelism within a crate. The default
   (`codegen-units = 256` for dev) is correct; only release builds use
   `codegen-units = 1` (for LTO).

5. **Use a generous cache size.** A full workspace build produces ~6-8 GiB
   of object files when all targets (lib, test, bench, example) are built.
   With 10 GiB the cache fills quickly and evicts recent units. 15 GiB
   provides a comfortable buffer. Do not set it above available free disk.

6. **Restart the sccache server after upgrading.** The server caches the
   rustc path and version at startup. After `rustup update` the version
   changes and old cache entries are invalidated (correct), but the server
   needs a restart to pick up the new binary location if it changed.

   ```sh
   sccache --stop-server  # or it restarts automatically on next cargo build
   ```

7. **For CI (future): use a shared remote backend.** The GitHub Actions cache
   backend (`SCCACHE_GHA_ENABLED=true`) or an S3/Redis store allows cache
   entries built on CI to be reused locally and vice versa. Relevant when
   the strict-local-only CI policy (ADR-009) is relaxed.

## Troubleshooting

### Build fails: `could not execute process sccache`

sccache is not on PATH. Install it (see above) or temporarily bypass:
```sh
RUSTC_WRAPPER="" cargo build
```

### Cache hit rate is 0% after reinstall

The sccache server may be caching against a different rustc binary.
```sh
sccache --stop-server
cargo build  # server restarts automatically with correct rustc path
```

### `sccache --show-stats` shows wrong cache dir

The server is running with stale config. Stop it, edit `~/.config/sccache/config`,
then let it restart:
```sh
sccache --stop-server
sccache --show-stats  # triggers restart and reads new config
```
